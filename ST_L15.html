Testing of Source Code Using Graph Models

Testing of source code using graph models involves representing a program through a control flow graph where each node represents a statement or block of statements and edges show the flow of control between them. To perform more effective white-box testing, the control flow graph is extended with information about variable definitions and uses. This extension allows testers to apply data flow coverage criteria, which help in identifying whether variables defined in the program are correctly used before being redefined or discarded.

In data flow testing, every variable undergoes two key actions: definition and use. A variable is said to be defined when it is assigned a value, and it is used when its value is read or referenced in an expression or condition. A def-use path is the sequence of program statements that connects a definition of a variable to its corresponding use without any redefinition of the same variable in between. Such paths are called definition-clear paths and form the basis of data flow testing.

The main criteria in data flow testing are all-defs coverage, all-uses coverage, and all-du-paths coverage. The all-defs criterion ensures that each definition of every variable reaches at least one of its uses, confirming that every assignment has an effect in the program. The all-uses criterion demands that each definition reaches all possible uses of that variable, ensuring that every situation where the variable is used has been tested. The all-du-paths criterion is the strongest because it requires every possible definition-clear path from each definition to each use to be executed at least once.

By applying these criteria, testers can detect logical errors such as unused definitions, incorrect variable updates, and missing computations. It helps to identify situations where data is defined but never used, or where a variable is used before being properly initialized. This method ensures that data dependencies in the program are thoroughly verified and that variable lifetimes are correctly handled.

It ensures that every path a variable can take from definition to use is checked, thereby improving the reliability of the software. Data flow testing using control flow graphs thus strengthens the overall structural testing process by combining both control and data aspects of the program.
